# Module Interfaces

**Document Version:** 1.0

**Status:** Draft

---

# 1. Purpose

This document specifies the interfaces between the modules of the Intent-Based Networking LLM Framework.

Rather than defining implementation details, this document describes the information exchanged among framework components.

Every module exposes a well-defined input and output, allowing the framework to remain modular, extensible, and independent of implementation technologies.

---

# 2. Processing Pipeline

The framework is composed of a sequence of independent modules.

Each module performs a single responsibility and communicates with the next stage through domain objects defined in the Core Domain.

```text
Dataset
    │
    ▼
Topology
    │
    ▼
Mininet Builder
    │
    ▼
Running Network
    │
    ▼
ONOS Collector
    │
    ▼
Topology
    │
    ▼
Knowledge Graph Builder
    │
    ▼
LLM Planner
    │
    ▼
Recommendation
    │
    ▼
Validation Engine
    │
    ▼
Validation Report
    │
    ▼
Execution Engine
    │
    ▼
Updated Topology
```

# 3. Module Interfaces

## 3.1 Topology Module

### Responsibility

Defines the unified network representation adopted by the framework.

### Input

None.

### Output

Topology

### Consumed By

- Mininet Builder
- Knowledge Graph Builder
- LLM Planner
- Validation Engine
- Execution Engine

---

## 3.2 Mininet Builder

### Responsibility

Creates a running Mininet topology from a Topology object.

### Input

Topology

### Output

Running Network

### Dependencies

- Mininet

---

## 3.3 ONOS Collector

### Responsibility

Discovers the current network state through the ONOS REST API and converts it into the unified topology model.

### Input

Running Network

### Output

Topology

### Dependencies

- ONOS REST API

---

## 3.4 Knowledge Graph Builder

### Responsibility

Transforms a Topology object into a graph representation stored in Neo4j.

### Input

Topology

### Output

Knowledge Graph

### Dependencies

- Neo4j

---

## 3.5 LLM Planner

### Responsibility

Generates network improvement recommendations based on the current topology.

### Input

Topology

### Output

Recommendation

### Dependencies

- Large Language Model

---

## 3.6 Validation Engine

### Responsibility

Performs deterministic validation of recommendations.

The Validation Engine guarantees that no recommendation reaches the execution stage without satisfying every validation rule.

### Input

- Topology
- Recommendation

### Output

Validation Report

---

## 3.7 Execution Engine

### Responsibility

Applies approved recommendations to the running network.

Rejected recommendations never reach this module.

### Input

Validation Report

### Output

Updated Topology

---

## 3.8 Metrics Engine

### Responsibility

Computes structural and operational metrics before and after network modifications.

### Input

Topology

### Output

Metrics

---

# 4. Data Flow

The following domain objects are exchanged among framework modules.

| Domain Object | Producer | Consumer |
|----------------|----------|----------|
| Topology | Topology Module | All modules |
| Recommendation | LLM Planner | Validation Engine |
| Validation Report | Validation Engine | Execution Engine |
| Metrics | Metrics Engine | Validation Engine |
| Updated Topology | Execution Engine | Experiment Manager |

---

# 5. Interface Principles

Every interface follows the same architectural principles.

## Single Responsibility

Each module performs only one task.

---

## Technology Independence

Business logic remains independent from infrastructure technologies.

---

## Loose Coupling

Modules communicate only through domain objects.

Direct module-to-module dependencies are avoided.

---

## Deterministic Validation

Recommendations generated by external systems are never executed directly.

Every recommendation must first be validated.

---

## Reproducibility

Identical inputs shall always produce reproducible experimental workflows.

---

# 6. Future Interfaces

The architecture allows additional modules without modifications to existing interfaces.

Possible future modules include:

- Policy Engine
- Intent Translator
- Security Analyzer
- Digital Twin Adapter
- Containerlab Adapter
- GNS3 Adapter
- Multi-Domain Coordinator

The interface contract remains unchanged regardless of the underlying implementation technology.
